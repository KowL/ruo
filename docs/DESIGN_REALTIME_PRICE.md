## 3. 核心接口设计

### 3.1 东财批量实时行情接口

**接口**: `http://push2.eastmoney.com/api/qt/ulist.np/get`

**参数说明**:

| 参数 | 类型 | 说明 |
|------|------|------|
| `secids` | string | 股票代码列表，格式 `市场.代码`，如 `0.000001,1.600000` |
| `fields` | string | 返回字段，用逗号分隔 |
| `fltt` | int | 浮点数精度，`2`=2位小数 |
| `invt` | int | 数据格式，`2`=JSON |

**市场代码映射**:

| 前缀 | 市场 | 东财代码 |
|------|------|----------|
| 0, 3, 2 | 深圳 | `0` |
| 6, 688 | 上海 | `1` |
| 4, 8, 9 | 北京 | `0` (北交所特殊处理) |

**关键字段映射**:

| 字段 | 含义 | 示例 |
|------|------|------|
| `f12` | 股票代码 | `000001` |
| `f13` | 市场代码 | `0`=深圳, `1`=上海 |
| `f14` | 股票名称 | `平安银行` |
| `f2` | 最新价 | `10.50` |
| `f3` | 涨跌幅(%) | `2.35` |
| `f4` | 涨跌额 | `0.24` |
| `f5` | 成交量(手) | `123456` |
| `f6` | 成交额(元) | `123456789` |
| `f17` | 开盘价 | `10.30` |
| `f18` | 昨收价 | `10.26` |
| `f15` | 最高价 | `10.60` |
| `f16` | 最低价 | `10.20` |
| `f20` | 总市值 | `2000000000` |
| `f21` | 流通市值 | `1500000000` |

### 3.2 雪球接口（备用）

**接口**: `https://stock.xueqiu.com/v5/stock/batch/quote.json`

**认证方式**:
- 需要有效的 `xq_a_token` Cookie
- 通过访问首页 `https://xueqiu.com` 获取 Token
- Token 有效期约 2 小时，需要定时刷新

**Token 管理策略**:

| 机制 | 说明 |
|------|------|
| **自动获取** | 首次调用时自动访问雪球首页获取 Token |
| **定时刷新** | Celery 每 30 分钟强制刷新 Token |
| **失效检测** | 检测 401/403 错误码自动重试 |
| **重试机制** | 获取失败时最多重试 3 次，指数退避 |

**Token 刷新流程**:
```
首次调用 / Token 过期
    ↓
清除旧 Cookie
    ↓
访问 https://xueqiu.com (带完整请求头)
    ↓
提取 xq_a_token
    ↓
保存 Token 和时间戳
    ↓
正常请求数据
```

**特点**:
- 需要 Cookie/Token 认证
- 支持批量查询（最多30只/次）
- 返回字段较丰富
- 限流相对严格

## 4. 缓存策略

### 4.1 多级缓存架构

```
L1: 内存缓存 (MarketDataService)
    ├── 实时价格: 5秒 TTL
    ├── 批量行情: 5秒 TTL  
    └── 盘口数据: 3秒 TTL

L2: Redis 缓存
    ├── 实时价格: 30秒 TTL
    ├── 股票详情: 1小时 TTL
    └── 搜索建议: 24小时 TTL

L3: 数据库 (PostgreSQL)
    └── 历史K线: 持久化存储
```

### 4.2 缓存更新策略

| 数据类型 | 缓存时间 | 更新触发条件 |
|----------|----------|--------------|
| 实时价格 | 5秒 | 新请求且缓存过期 |
| 批量行情 | 5秒 | 定时任务或请求触发 |
| 盘口五档 | 3秒 | 实时性要求高，短TTL |
| 股票基础信息 | 1小时 | 低频变化 |
| 搜索建议 | 24小时 | 日终更新 |

## 5. 熔断降级机制

### 5.1 熔断器设计

```python
class CircuitBreaker:
    """
    熔断器状态机
    
    CLOSED    → 正常，请求通过
    OPEN      → 熔断，直接返回降级数据
    HALF_OPEN → 半开，尝试恢复
    """
```

**触发条件**:

| 状态 | 触发条件 | 行为 |
|------|----------|------|
| CLOSED → OPEN | 连续失败 >= 5次 | 熔断，切到备用源 |
| OPEN → HALF_OPEN | 熔断持续 60秒 | 尝试恢复 |
| HALF_OPEN → CLOSED | 连续成功 >= 3次 | 恢复主源 |
| HALF_OPEN → OPEN | 任一失败 | 重新熔断 |

### 5.2 降级策略

```
主数据源(东财)  ──►  失败 ──►  备用数据源(雪球)
     │                      │
     ▼                      ▼
  正常数据               降级数据
  + 时间戳               + 降级标记
                         + 缓存数据
```

**降级响应示例**:

```json
{
  "symbol": "000001",
  "price": 10.50,
  "changePct": 2.35,
  "source": "xueqiu",      // 数据来源
  "degraded": true,        // 降级标记
  "cached": true,          // 是否缓存数据
  "timestamp": "2026-03-06T14:30:00Z"
}
```

## 6. 错误处理与重试

### 6.1 错误分类

| 错误类型 | HTTP状态码 | 处理策略 |
|----------|------------|----------|
| 网络超时 | - | 指数退避重试，最多3次 |
| 限流(429) | 429 | 立即熔断，切换备用源 |
| 服务不可用 | 5xx | 指数退避重试，最多3次 |
| 数据格式错误 | - | 记录日志，返回None |
| 无效股票代码 | - | 直接返回错误，不重试 |

### 6.2 重试策略

```python
# 指数退避
retry_delays = [0.5, 1.0, 2.0]  # 第1次0.5s，第2次1s，第3次2s

# 触发条件
- 网络异常 (ConnectionError, Timeout)
- HTTP 5xx 错误
- 数据解析失败 (可重试)
```

## 10. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 东财接口变更 | 高 | 监控响应字段，异常时自动降级 |

| 雪球也限流 | 中 | 使用缓存数据，返回降级标记 |
| 缓存雪崩 | 高 | 随机TTL，多级缓存，熔断保护 |
